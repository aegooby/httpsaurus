Authentication

Authentication is the way overly complicated way of saying what we allow which users to do. It's like parental controls, which means that we treat users like irresponsbile 5 year olds. Considering the IQs we're dealing with, it's best that it stays that way.

JSON Web Tokens#

JWTs are used for authentication. I'm not gonna write too much about it because the good folks at that link probably wrote a lot of informative stuff. Hold on. When a user logs in and gives us details that we would never misuse, like their passwords credit card information, we create an object that has all the details we need to check who they are. For example,

{  "id": "nodes:users:*:70af6899-189c-470c-8bd7-f14ffc3247b8",  "email": "mu@gabwe.com",  "roles": [    "Admin",    "Despot",    "Deceased"  ]}
note

The values in this object don't represent the entirety of what's stored in the database, just an example minimum amount needed for authentication. See the GraphQL schema section for more information.

Access token#

When the user logs in, assuming they've been a good boy and their login details are right, the server sends them an encrypted version of this object (the access token) which is stored on their end. Then, when they make a request, they set the Authorization header to include the token. Well, we set it for them, because they're definitely not smart enough to do that themselves.

Authorization: "Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIi..."

On the server, we use a magical library that checks if the token is valid, and we are really trusting them to get this right, otherwise anyone could hack into our system. If it is, then we let them do whatever they were supposed to do.

Refresh token#

The only problem is, when the user refreshes the page, the access token disappears. There are apparently very good security reasons for this that people have written surprisingly long essays about. I find it a bit much so I won't link it. That's right, suck my cheeks you long essays.

That's why, when the user logs in, we also send them a refresh token, which encrypts the same information but with a different key, and is stored as a cookie. No, we don't ask for user consent for the cookies. We force that shit on them and they just have to take it.

Since the encryption key on the refresh token is different, it can't be used to access restricted information, but can be used to request a new access token for the user. This is done by making a request to "/jwt/refresh" with the cookie set, which is done automatically when the user refreshes. This token expires after 7 days, so if the user waits that long, they have to log in again.

Decorators#

@Auth.authenticated()#

Restricted content on the server is done through GraphQL, so if one of our little users tries to be a bad boy and access stuff that they shouldn't, like pulling on a fire alarm in a gathering when there's no fire and making their dad very angry and giving mom a headache, then we restrict them. This is done using the @Auth.authenticated() decorator with a condition function. For example, if we only want users to see the user profile page that belongs to their user while logged in,

server/resolvers.ts
class Query implements QueryResolvers<Context> {  // ...  @Auth.authenticated(function (payload: UserJwt, args: { id: string }) {    return payload.id === args.id;  })  async readUserProfile(    _parent: unknown,    args: { id: string },    _context: Oak.Context,    _info: graphql.GraphQLResolveInfo,  ) {    // ...    return user;  }  // ...}

Looks complicated, but to an enlightened soul like yours truly, all this does is it checks that the token the user sends with their request matches the user id of the profile that they're requesting.

@Auth.rateLimit()#

Sometimes, someone isn't being a bad boy by not being logged in, but instead being a fucking asshole by spamming our shit. When someone does this, we use the @Auth.rateLimit() decorator and pass it a limit of the number of times they can do it, and an expiry time in seconds. For example, if we want someone to only be able to do a certain mutation a maximum of 5 times in 10 minutes,

server/resolvers.ts
// ...class Mutation implements MutationResolvers<Context> {  // ...  @Auth.rateLimit({ limit: 5, expiry: 60 * 10 })  async doSomethingThatCostsUsALotOfMoney(    _parent: unknown,    args: {},    _context: Oak.Context,    _info: graphql.GraphQLResolveInfo,  ) {    // ...  }  // ...}
